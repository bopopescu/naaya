NaayaThesaurus

 

 

  1. General description

This product is used in the Naaya architecture to store and handle a thesaurus.

 

  1. Prerequisites

In order to use this product, you should have a Naaya compliant site (with TextIndexNG installed).

 

  1. Installation

Installation of NaayaThesaurus is the same as for any other Zope product. You

just need to untar the package into the Products directory of your

Zope installation. Consult the Zope documentation if you need further

information.

 

For Windows, you can use WinZip or any similar utility to open the

NaayaThesaurus.tar.gz package (where NaayaThesaurus.tar.gz is the

name of the NaayaThesaurus package you downloaded) and extract all the files

into the appropriate directory.  For Linux and other Unix systems, cd

to the appropriate directory and do a tar xvzf NaayaThesaurus.tar.gz

(where NaayaThesaurus.tar.gz is the name of the NaayaThesaurus package

you downloaded).

 

After installing the NaayaThesaurus, restart Zope.

 

  1. Usage

Open the Zope management interface in a web browser (e.g. go to

http://your-zope-server/manage) and into you Naaya compliant site instance select the 'Naaya Thesaurus’ option from the pull-down menu labeled 'Select type to add...' and create a new NaayaThesaurus instance.

 

Inside your NaayaThesaurus instance you should find the following objects:

-  alt_terms (Alternative Terms)

-  concept_relations (Concept Relations)

-  concepts (Concepts)

-  definitions (Definitions)

-  scope_notes (Scope Notes)

-  source (Source)

-  terms (Terms)

                        -  theme_relations (Theme Relations)

-  themes (Themes)

-  thesaurus-labels.rdf (Labels and definitions)

-  thesaurus-relations.rdf (Concept relationships)

-  thesaurus-themes-labels.rdf (Theme labels)

-  thesaurus-themes-relations.rdf (Themes relationships)

-  thesaurus_catalog (ThesaurusCatalog)

-  thesaurus_css (Naaya Thesaurus CSS)

 

            The objects with  icon are data storage modules and will be explained in detail into chapter 5, as they represent the core of the application.

            The objects with  icon are customizable Python Scripts for SKOS/RDF export. For example if you want the exports to be shown in a specific way just edit those objects and write the desired code. The necessary functions (getters) are found in the code, just write your own code. (This will reflect on exported files on ‘RDF export’ tab found on end user interface)

            The object with  icon is the thesaurus catalog, here are indexed all data added into thesaurus.

            The object with  icon is the customizable end user CSS file. Just edit it to make any changes on the end user display.

 

  1. Data modules (marked with  icon)

The data inside thesaurus is linked with complex relations, thus an order is required on adding it in order to maintain its consistency because of the constraints found on adding. (e.g. you cant insert an theme translation for a given theme id if that theme id does not exist). After adding data, more constraints are found (explained below) on edit and delete any record.

 

On any data module you can find the ‘Properties’ tab from where you can set the title of the data module (this property is reflected only on Zope management interface), the ‘Management’ tab from which you can handle the data stored inside (add/edit/delete), the ‘Statistics’ tab where you can see information related to its data and the ‘Undo’ tab from where you can undo any action you made on the current object. More, on some relevant modules a ‘Import’ tab can be found from where a bulk, XML (SKOS/RDF) based, import can be done.

 

5.1 Concepts module

This is the first module to be filled with data. You can manually add from ‘Management’ tab or import a XML from ‘Import’ tab. From Management/Import tab you can freely add/import any concept ids you desire, no constraints on this module on add operation.

From ‘Management’ tab only a value for the concepts id is required (e.g. 77) on add.

From ‘Import’ tab you should provide a XML like the one found in ‘../doc/xmls/relation_theme.xml’. The XML structure should be like this:

 

<rdf:Description rdf:about="http://www.semide.org/thesaurus/concept/3">

                <thesaurus:theme rdf:resource="http://www.semide.org/thesaurus/theme/0002"/>

                <thesaurus:theme rdf:resource="http://www.semide.org/thesaurus/theme/0001"/>

</rdf:Description>

 

At this point a concept with id ‘3’ will be added on ‘Concepts’ module and two theme-relation records one with concept id ‘3’ and theme id  ‘0002’ and another with concept id ‘3’ and theme id ‘0001’ will be added on ‘Theme Relations’ module. A concept should belong at least to a theme (in the above example it belongs to two themes). After importing the ‘../doc/xmls/relation_theme.xml’ an information message will be displayed like below:

 

Information

File imported successfully.

Concepts added: 3

ThemeRelations added: 4

 

which inform you about the data added. If the imported XML if broken no actions will be done and a message like below will be displayed:

 

Errors

Parsing error. The file could not be parsed.

 

 

 

5.2 Theme Relations module

This module contains no ‘Import’ tab since the data in here is imported from the Concepts module. At this point you should find 4 records (assuming you previously imported ‘../doc/xmls/relation_theme.xml’) on Management tab. Further more you can add more records based on the concept ids you have in Concepts module. If you try to add a non existing concept id an error message will be displayed informing you that the respective concept id does not exist so no relation can be defined.

Adding e new (unique) theme id will practically create a new theme id. No constraint on theme id field, you can create as many as you want.

 

5.3 Themes module

On this module are stored the themes translations, from Management tab you can add using an existing theme id (declared on Theme Relations module) and selecting the desired language any translation.

From Import tab you can import data using XMLs with the structure as described below:

 

<skos:Concept rdfs:label="AGRICULTURE" rdf:about="http://www.semide.org/thesaurus/theme/0001">

</skos:Concept>

 

If you import the XML found on ‘../doc/xmls/themes[fr].xml’ you will add  French translations for existing themes (those with ids 0001 and 0002).

 

On Import tab there is a language combo, from there you should select a language

 in the case on the imported XML the language is not declared (e.g. xml:lang="fr"), if one is declared in the XML the language selection will be ignored. (this apply to all language combos found on Import tabs on the other modules)

 

5.4 Concept Relations module

On this module you define the relations (broader, narrower and related) between concepts. The only constraint on adding is that you should provide existing concept ids for ‘concept id’ and ‘relation id’ fields. Note that at this point the consistency of the relations you add (e.g. you can make a concept be broader and narrower to a same concept) is your responsibility.

On Import tab you should provide an XML with the structure as described below:

 

                <skos:Concept rdf:about="http://www.semide.org/thesaurus/concept/1">

                                <skos:narrower rdf:about="http://www.semide.org/thesaurus/concept/2"/>

                                <skos:related  rdf:about="http://www.semide.org/thesaurus/concept/3"/>

                </skos:Concept>

 

                <skos:Concept rdf:about="http://www.semide.org/thesaurus/concept/2">

                                <skos:broader rdf:about="http://www.semide.org/thesaurus/concept/1"/>

                </skos:Concept>

 

                <skos:Concept rdf:about="http://www.semide.org/thesaurus/concept/3">

                                <skos:related  rdf:about="http://www.semide.org/thesaurus/concept/2"/>

                </skos:Concept>

 

If you import the XML found on ‘../doc/xmls/ relations.xml’ four relations will be added.

 

5.5 Terms module

      From here you can add concepts translations and their associated source ids.

      From Import tab you can import XML with the structure as shown below:

 

<skos:Concept rdf:about="http://www.semide.org/thesaurus/Concept/2">

<skos:prefLabel>acidification

</skos:prefLabel>

<rdf:source>source de limite 3</rdf:source>

<skos:altLabel>acidification alternative</skos:altLabel>

<skos:scopeNote>note de portée</skos:scopeNote>

<skos:definition>Opération qui consiste à introduire de l'acide dans des roches carbonatées telles que calcaires, dolomies, sables calcaires ou grès pour accroître leur perméabilité en dissolvant une partie des constituants</skos:definition>

<thesaurus:def_src>source de définition 1</thesaurus:def_src>

</skos:Concept>

 

Importing the XML found on ‘../doc/xmls/terms[fr].xml’ you will add 3 concept translations, one alternative translation, one definition, one scope note and four sources, all in French.

 

5.6 Alternative Terms, Definitions, Scope Notes and Source modules

None of these module have an Import tab, the data for them will be imported via Terms module Import tab. All the data for these modules is related to the concept description and can be added from the Management tabs as described on the other modules.

 

 

6.      Delete/Update constraints and triggers

On Theme Relations and Concepts modules there are constraints/triggers on delete and edit. On Theme Relations for example, deleting all records that make reference to a theme id will automatically trigger the deletion of all its translations from Theme module. Also deleting all references of a concept to any theme id will display a warning that inform you that a concept was left without any theme association.

Similar for Concept module, when deleting a concept all its references from Concept Relations, Terms, Definitions modules … etc will be deleted.
